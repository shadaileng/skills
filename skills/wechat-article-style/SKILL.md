## 核心定位

> **"人话版"技术文章——专业但不端着，有痛有解有温度。**

不是干巴巴的文档，而是一个懂你的人在跟你聊天，帮你把复杂问题拆解成能听懂的话。

---

## 标题规范

### 主标题

- 格式：`<emoji> <中文长标题，15-25字>`
- 用问句/感叹句制造共鸣，避免陈述句
- 示例：

```
🤔 你被身份验证折磨过吗？
🎯 这篇"避坑指南"请收好
⚡ Redis 缓存总是失效？这几个坑你一定踩过
```

### 段落标题（二级标题）

- 格式：`## <emoji> <中文标题>`
- 每个段落标题独立承载一个话题点
- 常用 emoji 体系：

| 场景 | 推荐 emoji | 用途 |
|------|-----------|------|
| 提问/引发思考 | 🤔 | 开篇、引入问题 |
| 收获/目标 | 🎯 | 读完你会收获 |
| 要点/重点 | 📌 | 需要注意的事项 |
| 原理/拆解 | 🧰 | 架构、原理分析 |
| 实战/操作 | 🛠️ | 代码演示、步骤教程 |
| 警告/避坑 | ⚠️ | 踩坑点、注意事项 |
| 建议/思考 | 💡 | 决策指南、选型建议 |
| 总结/礼物 | 🎁 | 文末总结 |
| 行动/号召 | 🔥 | CTA、互动引导 |

### 三级标题（小节标题）

- 格式：`### <emoji> <小标题>`
- 用 🔹 / 🔸 / 💎 作前缀区分层级
- 示例：`### 🔹 数据库迁移一定要手动检查`

## 文章结构（7 段式模板）

每篇文章严格按以下结构组织：

### 1. 金句引言（Blockquote）

```markdown
> <一句话点题，30-50字，概括全文核心观点>
>
> <第二句展开，补充背景或痛点>
>
> <第三句给出承诺或期望>
```

**要求**：3-4 行，不超过 100 字。用 `>` 引用块格式，加粗关键词。

### 2. Hook 段落（痛点切入）

以读者的痛点/困惑开场，制造共鸣：

```
## 🤔 你被XXX折磨过吗？

<用2-3句话描述痛点场景，用"你"直接对话读者。>

那一刻我就知道，XXX这事儿，必须得有一套"开箱即用、又不容易犯蠢"的方案。
```

**要点**：
- 用"你"直接对话
- 描述具体场景而非抽象概念
- 用情绪词："头秃"、"翻车"、"一脸懵"、"大型面向复制粘贴编程现场"

### 3. 作者人设介绍

紧接 Hook 段落之后，用 2-3 行建立信任：

```
<emoji>我是爱折腾的**<作者名>**，喜欢研究**<领域>**的各种实践，
热爱分享**<风格描述>**，也享受用代码写出各种**<特色>**的快乐。

如果你也在技术这条路上向前走，关注我，愿我们能**彼此陪伴，一起成为更好的自己** 🌱
```

### 4. 收获清单

```markdown
## 🎯 这篇文章能帮你解决什么

**读完你会收获：**

🔹 <收获1，一句话>
🔹 <收获2，一句话>
🔹 <收获3，一句话>
🔹 <收获4，一句话>
```

3-5 条，每条不超过 25 字。

### 5. 正文主体（实战/原理/避坑）

正文至少包含以下 2-3 个模块：

#### 原理拆解模块

```
## 🧰 核心原理：它到底是怎么运作的？

<用比喻/类比解释复杂概念>
<分层讲解核心组件>
```

**比喻技巧**：
- 餐厅出餐口 → 路由系统
- 乐高拼装 → 组件组合
- 打补丁 → 临时修复
- 婊子 → 不要用

#### 实战教程模块

```
## 🛠️ 实战演示：从零搭一个XXX

<步骤化教程，代码块 + 中文口语注释>
```

#### 避坑专题模块

```
## ⚠️ 再说几个容易翻车的点

### 🔹 <坑1>
### 🔹 <坑2>
### 🔹 <坑3>
```

**要求**：每个坑点写清"是什么 → 为什么会出问题 → 怎么解决"

#### 决策指南模块

```
## 💡 我到底该不该用XXX？

<主观但有依据的建议>
<适用场景 vs 不适用场景>
```

### 6. 总结收尾

```markdown
## 🎁 总结一下

<用2-3句话回顾全文要点，升华主题>

<最后给出情感升华或鼓励>
```

### 7. CTA 行动号召

```markdown
如果你觉得这篇"人话版"指南有点用，别藏着掖着，
**点赞收藏加关注**，防止下次想找的时候找不到～

你在XXX上还踩过哪些坑？评论区唠一唠，一起避坑才是真朋友 🎯
```

## 写作风格

### 语气规则

| ✅ 要这样 | ❌ 不要这样 |
|-----------|------------|
| "咱们来看看" | "本文将介绍" |
| "你可能会问" | "读者可能会有疑问" |
| "哈哈，还没有" | "尚未完成" |
| "坑踩过的人都懂" | "这是一个常见问题" |
| "别急，往下看" | "详见下文" |
| "说实话" | "客观而言" |
| "说人话就是" | "简而言之" |

### 网络化表达词库

**痛点词**：头秃、踩坑、翻车、一脸懵、劝退、打补丁、缝缝补补、大型复制粘贴现场

**正面词**：闭眼用、开箱即用、一键搞定、优雅、丝滑、香

**连接词**：好，咱们、说回来、更重要的是、更要命的是、最后啰嗦一句

**过渡词**：接下来重点来了、是不是以为这样就Ok了？哈哈还没有、有一点要特别注意

### 人称与视角

- **主视角**：第一人称"我"（分享经验）、第二人称"你"（对话读者）
- **避免**：第三人称被动语态、"笔者"、"本文"
- **口吻**：像朋友聊天，不是写论文

## 代码块规范

### 注释风格

代码中的注释必须用中文口语化表达：

```python
# ❌ 错误：干巴巴的注释
# Initialize JWT strategy
# Get user from database

# ✅ 正确：有温度的注释
# 这里用异步引擎，否则和 FastAPI 的协程打架，你会看到奇怪的超时错误
# 用户注册后你想自动发邮件，就覆盖这个方法
```

### 代码块前的引导语

每段代码前加一句中文说明，而不是直接扔代码：

```markdown
工具的选择上，我认为顺手的才是最好的。既然选了 SQLAlchemy，咱们就用
`DeclarativeBase` + `SQLAlchemyUserDatabase` 的组合。

from fastapi_users...
```

### 踩坑点强调格式

在代码块之后、关键处用以下格式强调：

```markdown
🔹 **这里有几个踩坑点要再啰嗦一句**

- 数据库会话必须走异步引擎，否则...
- 这个要和模型里的 id 类型严格对应，不然...
- 自定义 UserManager 类是埋钩子的最佳入口...
```

## 排版规范

### 视觉分隔

- 段落之间用 `---` 分隔
- 每个大模块之间用 `---` 分隔
- 代码块与文字之间留空行

### 列表装饰

- 一级列表用 🔹（蓝色）
- 二级列表用 🔸（橙色）
- 重点强调用 💎（钻石）
- 禁止使用无装饰的 `-` 列表（太干）

### 加粗规则

- 关键术语加粗：**FastAPI Users**、**JWT**
- 观点/结论加粗：**中小型项目闭眼用**
- CTA 加粗：**点赞收藏加关注**
- 不要过度加粗，每段最多 2-3 处

### 引用块（Blockquote）

- 开篇金句用 `>`
- 正文中的引用用于强调"最后啰嗦一句"类的叮嘱
- 不要滥用，全文不超过 3 处引用块

## 自检清单

- [ ] 标题是否带 emoji？（必须）
- [ ] 开头是否有 3-4 行的 blockquote 金句？
- [ ] 第一段是否以痛点/疑问 hook 切入？
- [ ] 是否有作者人设介绍段落？
- [ ] 是否有"读完你会收获"清单？（3-5 条）
- [ ] 每个段落标题是否带 emoji？
- [ ] 代码注释是否用中文口语？
- [ ] 是否有避坑/翻车专题段落？
- [ ] 是否有决策指南（该不该用）？
- [ ] 结尾是否有 🎁 总结？
- [ ] 是否有 CTA 行动号召？（点赞收藏关注 + 评论引导）
- [ ] 是否有 `---` 视觉分隔？
- [ ] 列表是否用了 🔹/🔸 装饰？
- [ ] 全文语气是否口语化？（"咱们"、"你"、"哈哈"）
- [ ] 是否有个人签名/作者名？
